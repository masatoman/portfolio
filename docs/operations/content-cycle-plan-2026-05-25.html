3. 新設するもの (3 つだけ)

3-1. weekly-content-planner routine

新設 毎週月曜 11:00 JST / Claude Code /schedule

Phase処理
1. データ収集pv-quality 分類 + seo-integrated 分析 + pdca JSON + GA4/GSC CSV を読み込み
2. 前週検証前週ブリーフの結果を GA4/GSC で自動計測 → 「狙いが当たったか」評価
3. ブリーフ生成リライト 2 本 (B/E ランク優先) + 新規 3 本 (GSC 伸びクエリ優先) = 最大 5 本
4. 保存Supabase content_briefs テーブルに INSERT
5. 通知Slack に前週結果 + 今週ブリーフの要約を投稿

新設 火-土 毎日 02:00 JST / Claude Code /schedule

Step処理
1content_briefs から status=planned の最高優先度を 1 件取得
2brief_type に応じて新規執筆 or 既存リライト
3lint-article-claims --fix で fabrication チェック (今回実装済)
4git push → auto-article-merge で CI 通過後に merge
5status → published に UPDATE + 配信パイプライン投入

5. ブリーフの選定ロジック

リライト候補 (2 本/週)

優先ランク条件アクション
1B 重症PV ≥ 7 + 直帰 ≥ 85%構造改善 (冒頭/タイトル)
2E 機会損失GSC 表示 ≥ 5 + クリック 0タイトル / description 修正
3C 隠れた強PV ≤ 5 + 滞在 ≥ 150s内部リンク + SEO 強化

新規候補 (3 本/週)

優先ソース選定基準
1GSC 伸びクエリ表示数 前週比 +50% 以上 + 対応記事なし
2コンバージョン強化/contact に最も近い記事の隣接テーマ
3dev_journalsource_selector 基準 35/40 点以上

上限厳守

新規 3 本 + リライト 2 本 = 最大 5 本。候補不足なら無理に埋めない。
「書かない」判断もブリーフに記録。各記事に 「なぜ今週書くか」の数字根拠 を必須化。

6. 自己検証ループ (毎週月曜)

planner が毎週月曜に前週ブリーフの結果を自動評価:

評価条件
狙い通りPV ≥ 期待値 80% かつ GSC 表示あり
PV のみPV OK だが /contact 遷移 0
GSC のみ表示あり + クリック 0
不発全指標未達
想定以上PV ≥ 期待値 150%

この評価結果が 次週のブリーフ選定ロジック にフィードバック。
「不発」だったテーマ系統は次週の優先度を下げる。「想定以上」の系統は追加候補にする。

9. Phase 2 構想: ポートフォリオ自動最適化エンジン

週次コンテンツサイクル (Phase 1) が安定したら、ファネル全体 を自動最適化する Phase 2 に進化できる:

📊
ファネル計測

記事 → Works → /contact の遷移率を記事別に自動追跡

🔄
ケーススタディ自動生成

dev_journal + git history から Works ページのケーススタディを自動更新

🎯
求人市場連動

求人サイトのトレンドスキルを監視 → ポートフォリオの強調項目を自動調整

Phase 2 の仕組み (Phase 1 安定後)

層自動化計測
Discovery ← Phase 1: データ駆動コンテンツサイクル GSC 表示 / クリック / PV
Credibility 記事品質の自動最適化 (B ランクリライト) 滞在時間 / 直帰率
Relevance Works / ケーススタディの自動更新
求人市場トレンドとの連動		 記事 → Works 遷移率
Action CTA の自動最適化 (記事内 /contact リンク) /contact 到達率 / 問い合わせ数
Phase 1 (コンテンツサイクル) で「何を書くか」を自動化。
Phase 2 (ファネル最適化) で「書いたものをどう売上に繋げるか」を自動化。
両方揃って初めて「案件獲得マシン」が完成する。
