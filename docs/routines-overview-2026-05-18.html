🌙night-writer
軸 A 毎日 02:00 JST last: 5/18 02:04 ✅ next: 5/19 02:06 masatoman.net + Qiita + zenn + sns-automation

何をしている

  • dev_journal の直近 14 日エントリから記事ネタを 40 点スコアで採点 (35 点以上で採用)
  • masatoman.net マスター記事を新規生成 + 4 媒体 (Qiita / zenn / はてな / Dev.to / note) に adapted 版を即 git push
  • article_pipeline テーブルに Day+1〜+5 で publish スケジュール INSERT
  • 誇張 grep ('MRR ¥XXX' '転換率 X%' 等) で 0 件になるまで自己修正

メリット

  • 軸 A 量産モード (月 30+ 本生成) の中核、 push 込みで完全ゼロタッチ
  • 5 媒体展開で SEO + 流入両面カバー
  • 誇張禁止 grep + 一次情報主体 70%+ の publish-gate で品質下限を担保
  • fresh clone 仕様で git 状態のドリフトなし

改善点

  • prompt が 8000 字超で メンテコスト高、 STRATEGY.md スナップショットが固まっている → 古くなるリスク
  • カニバリチェックが `ls + slug 近似` で甘い、 意味的重複は検知不可
  • 5 媒体 push の どれかが失敗しても Slack 通知が「N 件成功」 のみで原因見えない
  • Phase 3-A (dev_journal 採用) と Phase 3 (新規ネタ) の境界が曖昧、 連日同テーマ生成事故が起きやすい

📄masatoman-weekly-auto-article
軸 A 毎週月曜 13:00 JST last: 5/18 13:03 ✅ next: 5/25 13:04 masatoman.net

何をしている

  • night-writer と「どっちが何を書くか」 の境界が曖昧 (両者とも dev_journal を見る、 重複生成リスク)
  • publish-gate 通過率の累積メトリクスなし → 「skip が多すぎて月 0 本」 でも気付けない

🐺masatoman-tuesday-article-watchdog
軸 A 毎週火曜 13:00 JST 未実行 (5/19 初回) next: 5/19 13:05 masatoman.net

何をしている

  • 直近 24h に merged された label:auto-article の PR を `gh pr list` で取得
  • 各記事 URL を curl して 200/404 + HTML 中の 'undefined' 'null' '[object Object]' をチェック
  • frontmatter title 空 / description 30-200 字外 / 本文 500 字未満 を異常判定
  • 異常検知時に draft:true 化した unpublish PR を作成 (auto-merge せず、 人間 review 待ち)

メリット

  • auto-article の事故 (テンプレートエラー / SSR 失敗) を 24h 内に自動巻き戻し
  • masatoman 手動 publish PR には触らない設計で誤検知ブロック
  • night-writer / weekly-auto-article の「攻め」 に対する「守り」 の routine、 量産モードの安全装置

改善点

  • 初回 5/19 13:00 JST がぶっつけ本番 → masatoman が結果を確認する手順を Calendar に未登録 (要対応 #3)
  • 'null' 検知は本文内に普通に登場する単語 (例: 「null 安全な」 解説記事) で誤検知しやすい、 文脈無視の grep
  • unpublish PR は draft のまま作成 = 通知されないと放置リスク、 Slack 通知が prompt に未記載
  • 異常検知の対象が「24h 内 merged」 のみ → 24h 超え後の遅延発覚事故は拾わない
