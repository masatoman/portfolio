// 01 / 背景: なぜ多軸化が必要か

ユーザー指摘 (2026-05-13): 「issue-finder は僕視点でイシュー度をつけてる? 僕視点というのは僕が DX で解決できる範囲で見てる?」

scoring.ts を厳密に確認すると、 既存スコアはすべて以下のどちらかで、 「masatoman が DX で解決できる範囲か」 を直接測る指標が存在しなかった:

  • issue_score = 業界全体での重要度 (頻度 × インパクト × 感情)
  • solvability_score = 業界一般の技術で解決可能か (masatoman 個人の手で、 ではない)
  • personalFit = 個人事業との「適合度」 (補助金 / 工務店ターゲット / 技術キーワードの存在)
  • opportunity_score = 上記の合成 (DX 視点の判定ではない)

つまり「アフター放置 (solvability=55)」 という値は、 「ANDPAD 等の業界ツールで概念的には解決可能」 という業界視点であって、 「masatoman が Next.js + Supabase + Claude API で実装可能か」 という個人視点ではなかった。 結果として「即動け」 ラベルも業界視点の重要度に基づくもので、 個人が今動くべきかとは別物だった。

// 02 / 設計原則: イシュー度の純粋性 + 視点フィルタ

原則 1: イシュー度は業界視点で純粋に保つ issue_score は「業界全体での痛点の深さ」 だけを測る。 解決者が誰か (masatoman / 中小企業診断士 / 補助金支援者) は混ぜない。
原則 2: 解決視点はフィルタとして後から掛ける 「DX で解決できるか」 「経営支援で解決できるか」 「補助金で解決できるか」 は別軸として記録し、 ランキングや UI でフィルタとして適用する。
原則 3: 視点は N 個に拡張可能な構造 将来 masatoman が中小企業診断士の資格を取れば「経営支援視点」 が増える。 IT 導入支援事業者になれば「補助金支援視点」 が増える。 同じパターンで shindansiFeasibility や subsidyFeasibility を追加できる。

// 03 / 現状の scoring 軸 (5 軸)

issue_score (業界視点)
「業界全体での頻度 × インパクト × 感情強度」。 解決者は問わない。
issue_score = (clusterSize / samplingTotal) × 1.5 × 100 + industryImpact + (emotion-50)/5
solvability_score (業界視点 / 技術的解決可能性)
「業界一般の SaaS / 技術で解決可能か」。 masatoman 個人の手では、 ではない。
[SKILL or 人手で 0-100 の直接入力]
personalFit (個人事業適合度)
補助金親和性 / 主顧客業界マッチ / ツール愚痴 / 技術キーワード / solv 高さ で +20 ずつ。 「やる意味があるか」 を測る。
personalFit = subsidyMatch(20) + industryMatch(20) + toolMatch(20) + techKeyword(20) + (solv≥65)(20)
competitorDensity (競合密度)
クラスタ内に ANDPAD / freee / kintone 等の固有名がいくつ出るか。 多いほど競合占有度高。
0 ツール = 15、 1 ツール = 40、 2 ツール = 65、 3+ ツール = 85
salesChannelDifficulty (販売チャネル難易度)
industryTags から業種別の売りにくさを判定。 工務店 = 80 (対面営業必須)、 個人事業主 = 35 (Web 完結)。
industryTags に応じた固定値 (30-85)

合成: opportunity_score = issue × 0.25 + solvability × 0.30 + personalFit × 0.20 + (100-competitor) × 0.15 + (100-salesDiff) × 0.10

// 04 / 抜けていた軸: DX 解決可能性

solvability_score と何が違うか:

軸聞く相手例: アフター放置
solvability_score業界一般 (SaaS 業界全体)55 (ANDPAD 等のアフター管理機能で技術的には可能)
dxFeasibility (新)masatoman の手60 (機能実装は可能、 ただし入力ハードルは技術じゃ越えられない)

solvability=55 と dxFeasibility=60 は近い値だが、 意味が違う。 例えば「施工品質不良」 では solvability=25 (業界として品質検査 SaaS は存在しない) / dxFeasibility=10 (masatoman の手では特に無理) と差が出る。

// 05 / 新設計: dxFeasibility 軸追加

dxFeasibility (masatoman 視点 / DX 解決可能性) NEW
「masatoman の技術スタック (Next.js + Supabase + Claude API + Vercel + Resend) で実装+運用できるか」 を 0-100 で。
[人手で 0-100 の直接入力。 将来 LLM で自動推定可能]

値域の意味

値域意味
90-1001〜2 週間で MVP 可能、 既存 lab tools と接続可能
70-891〜2 ヶ月で実装可能、 既存スタックで完結
50-69実装可能だが運用負荷高い or 周辺技術習得必要
30-49技術スタックの大幅拡張必要 (ネイティブアプリ等)
10-29DX で部分支援は可能だが本質解決は別領域
0-9DX では解決不可能 (規制・物理・体制・体質問題)

新合成スコア: dxOpportunity

既存 opportunity_score を「DX で解決できる範囲に絞った場合の動くべき度」 にスケーリング:

dxOpportunity = opportunity_score × (dxFeasibility / 100)

dxFeasibility=0 → dxOpportunity=0 (完全に除外)
dxFeasibility=100 → 元の opportunity_score

既存スコアは触らない: issue_score / solvability_score / personalFit / opportunity_score はそのまま。 「業界視点ランキング」 は今まで通り見られる。 新しい「DX 視点ランキング」 が並列で追加される形。

// 06 / 拡張性: 視点軸の追加パターン

同じ構造で N 視点を追加可能。 将来の masatoman の役割変化に応じて軸を増やす:

shindansiFeasibility (中小企業診断士視点) 将来
中小企業診断士の資格取得後に追加。 経営支援 / 補助金申請代行 / 事業計画策定で解決できるか。
shindansiOpportunity = opportunity_score × (shindansiFeasibility / 100)
subsidyFeasibility (補助金支援視点) 将来
IT 導入支援事業者登録後に追加。 補助金スキームで顧客負担を圧縮できるか。
subsidyOpportunity = opportunity_score × (subsidyFeasibility / 100)
offlineFeasibility (リアル接触解決) 将来
建材屋 / 設計事務所経由のオフライン接点で解決できるか (memory [[komuten-channel-strategy]] の主戦場系)。
offlineOpportunity = opportunity_score × (offlineFeasibility / 100)
UI 側の使い方: ソートキーに dxOpportunity / shindansiOpportunity / subsidyOpportunity を追加。 「いま自分が動けるリソースは何か」 でフィルタを切り替える。 同じ業界クラスタを複数の視点で見られるようになる。
