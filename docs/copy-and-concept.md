## 1. 主顧客と訴求の前提

### 主顧客（メインの app/page.tsx が訴求する人物像）
- **属性**: IT に極端に弱い小規模事業オーナー、または 中堅幹部（現場監督・町工場・地元飲食店など）
- **語彙レベル**: 「モバイルデバイス」という単語の意味が分からない。電話・FAX・紙の伝票が日常
- **発注決定者**: 多くの場合は社長兼業務担当
- **副顧客**: 同じ会社の従業員・現場担当者（実際にツールを触る人）

### 訴求の3本柱
1. **困りごと → 解決した姿** で見せる
2. **金額の目安** を具体的に出す
3. **動画・スクショ** で「実物が動いてる」を見せる

### NG ワード（コピーから完全に排除）
- Next.js / Supabase / Anthropic API
- SaaS / MVP / API
- 認証 / DB / フロントエンド
- 「モバイルデバイス」「スマートフォン」（「スマホ」は OK）

### ポジショニングの原則
- **「ひとりで全部作れる職人」** を前面に出す。 企画 → 設計 → 制作 → 公開 → 運用 を 一気通貫で内製
- **「AI 使いの人」 ではない**。 AI は 問われたら答える程度に留める。 露出は profile/about/services から外し、 質問されたら『AI 機能の組み込みもできます』 と答える運用
- 理由: AI を表に出すと「AI で アプリを作って 遊んでる人」 印象に寄る。 主顧客 (社長 / 中堅幹部) は 流行り物に弱いと自覚し 距離を置きたがる傾向 → 職人ポジ で 信頼に倒す

## 2. ヒーローセクション

### 現状コピー
```
キャッチ: 残業の原因を、ITで撃退します。
サブ: 現場のお仕事に集中できるよう、面倒なパソコンの仕事をかわりに作ります。
ステータス: 新しいご相談をうけつけ中
プライマリ CTA: [ view works ]  → #works にスクロール
```

### 設計意図
- **「めんどくさい」** ではなく **「残業の原因」** を選んだ理由: 「めんどくさい」は読み手が「わかる」で止まる共感止まり。「残業の原因」は読み手が「うちの場合は何だろう」と頭の中で課題リストを回し始める = 行動を促す
- 社長視点でも「残業 = 人件費」と既に認識されているので、自動的に金勘定の文脈に乗る
- 「！」を使わず句点で終わることで、Y2K のクールな撃退感に統一

### 検討して却下した候補
| 候補 | 却下理由 |
|---|---|
| むずかしい話は、ぜんぶ こちらで。 | 主語と目的語が宙に浮いて「何屋？」になる |
| お店のしごと、スマホ1台に。 | 詩的だが「お店」限定で広がらない |
| ホームページも 業務アプリも、ぜんぶ こちらで。 | 何屋かは伝わるが平坦 |
| めんどくさいを、ITで撃退します。 | 抽象的で具体課題に繋がらない |
| 日々のしごとを、楽にする道具を 作ります。 | 動詞「作る」が弱い |
| あったらいいな。を、カタチに。 | 寄り添い系。 ブランド向きだが LP 行動換起力が弱い |
| 残業の原因を、ITで撃退します！ (「！」あり) | 営業 LP 寄りすぎ。 Y2K のクールトーンと不整合 |

## 9. 開いている論点（マーケ観点で再検討候補）

### コピー
- ヒーローの「残業」フレームは中小企業全般に効くが、「残業ゼロの家族経営」業種には刺さらない可能性
- サブキャッチ「現場のお仕事に集中できるよう、〜」は現場業のみを想起させる。事務職・サービス業の社長には弱いかも
- Services の価格目安（30/50/20万円〜）が、心理学的に高すぎ・安すぎないか未検証

### 構造
- 5 セクションは適切か。「お客様の声」「実績数字」「料金例」を足すと信頼上がるが、 hirao/bokoko 路線から外れる
- About が極端に短い（4行のみ）。社長視点では「相手はどんな人？」を確認したい欲求があるはず

### デザイン
- Y2K テイストは masatoman 個人の好み確定。 ただし IT 素人ターゲットの「信頼感」と相性が悪い可能性（「ゲーム会社っぽい」リスクは事前に伝達済み）
- 別路線（dark-grid の落ち着き / cinematic のエレガンス）への振り替え可能性

### 主顧客の妥当性
- 工務店現場監督がモデル化されているが、実際の案件発生母数を未検証
- B2B 実用デモが 工務店・中小企業 寄せに 偏っている。 飲食・美容・不動産 でも同様にデモ整備すべきか

### 心理学・マーケ観点で見直すべき要素
- 行動経済学: 「無料相談」を最初の 30 分と明示するアンカリング効果 → 既に CTA に反映済み
- 損失回避: 「残業の原因」フレームは損失回避を活用している
- 社会的証明: 顧客の声・導入実績がゼロ → 補強候補
- 互恵性: 「最初の30分無料」は互恵性ドライバー → 既に活用
- 権威性: profile.numbers の「5+ 年」「4 サービス自前運用」程度。 公開資料・登壇歴・受賞歴で補強候補

## 11. 補助金訴求の方針 (2026-05-09 追加)

### 現状フェーズ
- メイン (`app/page.tsx`) の Services と CTA の間に「補助金で 半額 になる かも。」 セクションを追加 (`Subsidies` コンポーネント、 Y2K アンバーカラー)
- Services 各カードに価格目安を再表示 (30万円〜 / 10万円〜 / 50万円〜) し、 各カード末尾に「→ 補助金で 半額〜2/3 補助の 可能性あり」 を アンカー リンク化 (#subsidies へジャンプ)
- 心理メカニクス: 価格 (30万円) を見せる → 補助金リンクで「実質半額」の可能性を示す → アンカリング + 損失回避 の二重作動

### 訴求できる補助金 / 訴求しない補助金
**訴求 OK** (パネル内に記載):
- 小規模事業者持続化補助金 (商工会議所)
- 業務改善助成金 (厚労省)
- 自治体ごとの IT 化補助金

**訴求しない**:
- IT 導入補助金 (IT 導入支援事業者経由) — masatoman が 支援事業者として 未登録のため 対象外。 disclaimer で明示

### masatoman の補助金関連ステータス (2026-05-09)
- ✅ **GビズIDプライム 取得済み** — 行政電子申請の本人認証アカウントを保有。 IT 導入支援事業者申請の前提要件はクリア
- ⏳ IT 導入支援事業者 登録 — 未登録。 公募タイミング待ち
- ✅ 他の補助金 (持続化 / 業務改善 / 自治体) は調べて提案できるリテラシーあり

### 申請業務のスタンス
- 「補助金コンサルになる」 リスクを避け、 申請書作成は **行政書士など 専門家を 紹介** する立場に留める
- パネル内 1 段落で「申請書類の作成が必要な場合は、 行政書士など 申請に強い 専門家を 紹介する 動きもできます」 と明示

### 将来の昇格パス (登録完了後)
1. disclaimer から「IT 導入補助金は 現状 対象外」 を削除
2. 「IT 導入支援事業者 登録済み」 のバッジ追加
3. 「IT 導入補助金で 最大 ◯◯万円 補助」 の 具体額訴求
4. 補助金パネルを Services 直下 → Hero 直下 に 格上げ検討
5. 顧客の補助金代理申請の動きを 1サービスとして 追加検討

### 検討して採用しなかった案
| 候補 | 不採用理由 |
|---|---|
| 「GビズIDプライム取得済み」 を 表示 | IT 素人顧客には何のことか分からない。 翻訳しても薄い。 IT 導入支援事業者登録完了まで 一括グレードアップ |
| 「30万円 → 補助金で実質10万円」 と 大きく数字対比 | アンカリング強いが、断定気味で誇大広告リスク。 「半額〜2/3 補助の 可能性」 と幅を持たせる方が安全 |
| 専用ページ `/subsidies` を作成 | 制度変動が激しく メンテコスト高。 現フェーズはメインのパネル 1 つで十分 |
| 「補助金狙い」 客向けに集客強化 | 値引き交渉客が増える。 質より量に傾くため、 現フェーズでは強化しない |
