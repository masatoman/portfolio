// 01 / 設計原則: ユースケース起点

原則 「どこに保存するか」 を決める前に、 「何の意思決定に使うか」 を全列挙する。 ユースケースの幅広さによって、 単層構造か 2 層構造かが決まる。

今回のヒアリングで判明したこと:

  • ユースケースは A〜D 全部 (1 つに絞らない、 仮説検証フェーズ全体の素材)
  • 取得頻度は スポット追加 (1 回限りでも cron 自動化でもない / コマンド 1 本で増やせる粒度)
  • 見る場所は 未定 (将来 HTML / Web ダッシュボード / ペライチ埋め込みなど複数想定)

この 3 つから自動的に出る結論: データを 1 つの形に正規化して貯めておき、 用途に応じてビューを切り替える構造でなければ詰む。 1 回ごとの JSON でも、 issue-finder の if_issues 単層でも、 ユースケース A (ペライチ引用) と C (機能優先順位) は両立しない。

// 03 / アーキテクチャ: 2 層モデル

LAYER 1
raw レビューストア (新テーブル)
Google Places API から取得した個別レビューを、 不変・追記のみで蓄積。 1 レビュー = 1 行。
役割: 一次情報の原本保存。 「同じデータから別の切り口」 を後から再実行可能にする。
用途: ユースケース A (引用) と D の集計部分はここを直接 SELECT。
追加方法: scripts/koumuten-review-collector.js --submit で取得と同時に INSERT。 重複は (place_id, publish_time, author_name) で吸収。
↓ SKILL で論点分解 + クラスタリング
LAYER 2
クラスタ化された痛点 (既存 if_issues 流用)
raw から論点を抽出し、 SKILL でクラスタリング → 既存 issue-finder の if_issues に投入。
役割: 「複数レビューを跨いだ共通の不満」 を浮かび上がらせる。 issueScore で動くべき度を算出。
用途: ユースケース B / C と D のクラスタ部分。 既存 matrix / cards UI で可視化済み。
追加方法: /issue-finder process SKILL で raw → クラスタリング → if_issues INSERT。 任意のタイミングで実行可能 (raw の追加と切り離されている)。
2 層に分ける本質的な理由: クラスタリングは「圧縮」 (5 件のレビュー → 1 つの痛点要約) なので、 引用本文の生々しさが失われる。 A (ペライチ引用) は生々しさが命なので raw が必須。 一方 B / C はクラスタの方が判断に使いやすい。 両方を別ストアで持つことで、 ユースケースごとに最適な解像度を選べる。

// 05 / Layer 2: クラスタリング (既存 if_issues 流用)

既存 issue-finder のスキーマと SKILL を そのまま流用。 perspective は komuten / 施主 (顧客側) で OK (前回ブリーフからの結論を維持)。

変更点は 1 つだけ:

  • SKILL の入力ソースが if_jobs.raw_input_text の文字列ブロックから、 raw テーブルへの SELECT に変わる。 具体的には、 if_jobs に投入する代わりに、 SKILL が直接 koumuten_reviews_raw から processed_in_cluster_job_id IS NULL の行を読んでクラスタリング → if_issues INSERT → raw 側に job_id を書き戻し。

これにより、 「同じ raw データから何度でも再クラスタリング可能」 になる (job_id を NULL に戻すか、 別ロジックを足せば)。

// 06 / データフロー

[1] 取得
node scripts/koumuten-review-collector.js --limit 20 --submit
  ├ Google Places API 叩く (キャッシュあれば省略)
  ├ レビューを正規化
  └ koumuten_reviews_raw に INSERT (重複は UNIQUE で吸収)

[2] (任意のタイミングで) クラスタリング
/issue-finder process (SKILL)
  ├ koumuten_reviews_raw WHERE processed_in_cluster_job_id IS NULL を SELECT
  ├ 論点分解 (ルールベース or LLM)
  ├ クラスタリング (LLM)
  ├ if_issues に INSERT (source_type='review')
  └ raw 側に job_id を書き戻し (再処理防止)

[3] 用途別に取り出す
A (ペライチ引用)  → koumuten_reviews_raw を SELECT で抜粋
B (ヒアリング仮説) → if_issues を perspective=施主 + source_type=review で SELECT
C (機能優先順位)   → if_issues + issueScore で並び替え
D (受託資料)       → 集計 SQL + クラスタ要約を組み合わせる

// 10 / 未決事項 (Phase 1 で詰める)

  • raw テーブル名: koumuten_reviews_raw でいいか / if_ プレフィックスに合わせて if_external_reviews 等にして他業界も同じテーブルで持つ可能性も。 → 今回は工務店専用で割り切る案を提示
  • 論点分解の精度: ルールベースで十分か LLM 必要か。 24 件で実測してから判断
  • 匿名化ルール: ペライチ引用時に投稿者名は隠す前提だが、 店舗名は出すか伏せるか。 工務店業界の場合は実名出すと風評リスクあり → 「都内 A 区の工務店」 表記に変える案
  • 取得対象拡張: 8 区 → 23 区全体 / 地方都市まで。 主顧客 (中小工務店) の声を厚く集めるなら地方データの方が高品質な可能性
  • API キーローテーション: 今回のチャットで Google Places API キーがログに露出。 動作確認終わったら Google Cloud Console で再生成

// 11 / 判定

このブリーフ v2 への判定 4 択:
  • ① OK、 Phase 1 (Layer 1 構築) から実装開始
  • ② 骨格 (2 層モデル) は OK だが、 Phase 1 の前にもう少し詰めたい論点がある → 内容を指定
  • ③ MVP マップ (A〜D) の優先順位を変えたい → どれを先 / どれを後にしたいか
  • ④ そもそも 2 層モデルではなく単層 (raw のみ / クラスタのみ) に変えたい
