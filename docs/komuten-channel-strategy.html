04localbiz の 構造的 ミスマッチ

localbiz の 設計 思想 = 「Web で 公開 されて いる イベント を 集めて スコアリング」。 でも 上記 の 通り 主顧客 が 接して いる ハブ は ほぼ 全部 Web に 載って いない:

  • 建材屋 の 営業マン → 対面 / LINE
  • 組合 の 機関紙・FAX 通信 → 紙
  • 信金担当者 の 集金 ついで の 世間話 → 対面
  • 飲み屋 で の 同業者 雑談 → 対面
  • 既存施主 から の 電話 → 電話

localbiz が 拾える のは 「補助金 説明会」 だけ。 それ以外 は ノイズ (Peatix の エンジニア 勉強会 / こくちーず の マーケ セミナー 等)。

★ 推奨

2 番手

何 を やる: localbiz の 構造 (Supabase / scoring / Tailscale) を 流用 して、 events テーブル を contacts テーブル に 改修。 イベント 一覧 で なく 建材屋・組合・信金担当者・既存施主 を 管理 する CRM に 変える。 「次回 訪問 予定」 「最後 に 会った 日」 「紹介 で 繋がった 案件」 等。

08残った 戦略 的 含意

含意 1: issue-finder は そのまま 維持 する。 「主顧客 が 集まる 場所 を 探す ツール」 と して は ミスマッチ だった が、 「主顧客 の ペイン を 発見 する ツール」 と して は 引き続き 有効。 ネット で 悲鳴 を 拾う ↔ 主顧客 が ネット 検索 し ない、 は 矛盾 し ない (悲鳴 を 書く のは 主顧客 ではなく 周囲 = 妻・家族・友人・OB・若手 職人)。
含意 2: 工務店向け SaaS / 受託 の 値付け (memory: saas-pricing.md の 4 パターン) は 販路 が アナログ で ある こと を 前提 に した もの に 修正 必要。 Web 完結 の セルフサーブ は 主顧客 と 噛み合わ ない。 「商談 → 見積 → 対面 商談 → 契約」 が 基本フロー に なる。
