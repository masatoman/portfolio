01全体像とデータフロー

商談相手の前で iPhone or Mac の ボイスメモ をタップして録音 → 終了後 inbox/ に投下 (Shortcuts で自動化) → launchd が検知 → whisper.cpp が文字起こし → Claude Code が要約整形 → Supabase に INSERT。 ここまで人手は 「録音ボタン押す」 と 「停止」 の 2 回 だけ。

[iPhone / Mac ボイスメモ録音] │ ▼ Shortcuts.app (自動コピー) [inbox/ に .m4a 投下] │ ▼ launchd WatchPaths 検知 [01_ingest.sh] ffmpeg で 16kHz mono wav 正規化 │ ▼ [02_transcribe.sh] whisper.cpp + Metal (large-v3-turbo-q5_0) │ → raw.txt + segments.json (タイムスタンプ付き) ▼ [03_summarize.sh] claude -p で要約 + 引用 5 件 + 固有名詞補正 │ → payload.json (project / category / kind / title / body / metadata) ▼ [04_insert.sh] supabase-js で dev_journal INSERT │ ▼ [Supabase dev_journal にレコード追加] → 後で記事化・戦略整理に活用
処理時間の目安: M4 Pro Metal で 約 7.6x realtime (実測 5/15、 JFK 11 秒 → 1.44 秒)。 60 分の長尺商談なら 約 8 分 で dev_journal に着地。 Rosetta 経由でこの速度なので体感不満なし。

02使い方 — 実運用フロー

商談前 (前日〜当日 直前)

前1ボイスメモ アプリを開いて録音準備

iPhone / Mac どちらでも OK。 オンライン面談 (北原氏など) の場合は Mac のボイスメモで画面の音声と自分の声を同時録音、 対面 (たましん訪問・友人ヒアリング) の場合は iPhone を机に置く。

前2(対面のみ) 録音許可を口頭で取る

「メモのため録音させていただいてもよろしいでしょうか」 で OK。 ローカル処理で外部送信しない旨も伝えると安心感が出る。

商談中

中1録音開始 → あとは話に集中

録音中の操作は不要。 紙メモは取っても取らなくてよい (録音が一次情報、 紙はあとから整理されるので)。 質問リストだけ手元に置いておく。

商談後 (終了 5 分以内に着手)

後1録音停止 + ファイル名を商談相手 + 日付に

例: 2026-05-22-kitahara-online.m4a。 ファイル名から title 自動生成されるので、 「日付 + 相手 + 形式」 を守ると後で検索しやすい。

後2inbox/ に投下

iPhone 録音 → Shortcuts.app の自動化で inbox/ にコピー (1 回設定済み)。 Mac 録音 → Finder で ~/home_work/portfolio/tools/voice-pipeline/inbox/ にドラッグ。

後3(オプション) project 手動上書き

Claude の project 自動推定がハズれそうな時のみ。 inbox/ に 2026-05-22-kitahara-online.project.txt を作って中身に 軸B-工務店 等と書く。 通常は不要 (Claude が文字起こしから推定)。

後45 分待つ → dev_journal を確認

launchd → 01_ingest → 02_transcribe → 03_summarize → 04_insert と自動進行。 完了通知は Mac 通知。 Supabase で確認:

SELECT title, body, metadata FROM dev_journal
WHERE category = 'meeting' ORDER BY created_at DESC LIMIT 1;

後日

日1記事化・戦略整理

article-source-selector skill が dev_journal を読むので、 商談メモが自動的に記事ネタ候補に上がる。 軸 B 関連は masatoman-article-rules skill の対象外だが、 友人ヒアリング知見は軸 A 記事ネタにも転用できる。

失敗してもメモは絶対に失わない: 録音 .m4a は inbox/ → done/ に常に残る。 文字起こしが失敗しても、 後で手動で whisper-cli を回し直すか、 Voice Memos の Apple Intelligence transcript を保険として使える。

03ツール選定と理由

役割採用不採用候補と理由
録音 macOS Voice Memos (Tahoe 標準) QuickTime / Audacity は手数多。 Tahoe の Apple Intelligence 自動文字起こし (日本語) が同時に走り、 句読点 backup として使える。 iPhone 録音も iCloud 同期で同パイプラインに乗る
文字起こし Plan A whisper.cpp + Metal、 large-v3-turbo Q5_0 M4 Pro で 60 分音声を約 8 分。 CLI 安定・JSON 出力枯れていて launchd と相性◎。 MLX Whisper は速いが CLI 周辺が薄い
文字起こし Plan B Voice Memos の Apple Intelligence transcript 句読点・話者切れが意外と良い。 Plan A と diff して固有名詞 (北原 / たましん / GビズID) を補完用に
監視・パイプ launchd LaunchAgent (WatchPaths) Hammerspoon は GUI 常駐必要。 fswatch は別途 daemon 化必要。 launchd 単体で macOS 再起動後も自動復帰・依存ゼロ
整形 LLM claude CLI (-p --output-format json) OpenAI / Gemini は従量課金。 Claude Code MAX 定額で十分。 prompt は固定 .md ファイルにして git 管理
INSERT supabase-js + session-recap skill schema 流用 MCP は対話セッション必須で cron 不可。 psql 直叩きは jsonb escape 地獄。 session-recap の payload schema を仕様書として固定すれば二重実装回避

04ファイル配置

~/home_work/portfolio/tools/voice-pipeline/
├── inbox/              # Shortcuts.app から .m4a を投下する着地点 (launchd 監視対象)
├── work/               # 16kHz wav / json / raw.txt 中間置き場 (.gitignore)
├── out/                # 要約 md / dev_journal payload json (.gitignore)
├── done/               # 処理済 .m4a 退避 (180 日 cron で自動削除)
├── prompts/
│   └── summarize.md    # claude -p の整形 prompt 本体
├── vendor/whisper.cpp/ # git clone (.gitignore)
└── bin/
    ├── 01_ingest.sh    # m4a → 16kHz mono wav (ffmpeg)
    ├── 02_transcribe.sh # whisper.cpp → raw.txt + segments.json
    ├── 03_summarize.sh # claude -p → payload.json
    ├── 04_insert.sh    # session-recap 仕様で dev_journal INSERT
    └── run_all.sh      # 01→04 orchestrator、 失敗で work/ に残し exit

~/Library/LaunchAgents/net.masatoman.voice-pipeline.plist  # WatchPaths = inbox/

各 shell script は「入力 1 個 → 出力 1 個 → ログ 1 行」 だけ。 巨大化させない。 どの段で落ちても work/ に途中ファイルが残り、 そこから再実行可能。

05dev_journal row 設計

session-recap skill の既存 schema に合わせる前提。 商談用に新 category=meeting を追加 (skill 側にまだ無ければ追記)。

カラム値の例備考
project軸B-工務店Claude が文字起こしから推定。 推定失敗時 $NAME.project.txt で手動上書き
categorymeetingsession-recap の decision/insight とは別軸の新カテゴリ
kindnegotiation / interview / field-visit商談 / ヒアリング / 現場訪問 の区別
title2026-05-22 北原氏オンライン面談日付 + 相手 + 形式 (ファイル名から自動)
body3 部構成 markdown① Claude 要約 300-500 字 (決定 / 宿題 / 温度感) ② 引用 5 件 (タイムスタンプ付き) ③ <details> 内に全文
metadatajsonb{ audio_path, duration_sec, transcribe_engine, model_quant, recorded_at, source: "voice-memos" }
「全文だけ」 だと検索性ゼロ、 「要約だけ」 だと固有名詞ロスト → 両方併記が必須。 Supabase の Full Text Search で全文ヒット、 LLM 食わせは要約使用、 という二段使いが効く。

06セットアップ手順 (実コマンド)

A. 依存ツール

# cmake と node を install (ffmpeg, jq は既存)
brew install cmake node

B. ディレクトリ作成

mkdir -p ~/home_work/portfolio/tools/voice-pipeline/\
{inbox,work,out,done,prompts,bin,vendor}

C. whisper.cpp ビルド (Metal-only)

cd ~/home_work/portfolio/tools/voice-pipeline/vendor
git clone --depth 1 https://github.com/ggerganov/whisper.cpp
cd whisper.cpp
cmake -B build -DGGML_METAL=1 -DWHISPER_COREML=0
cmake --build build -j --config Release
bash ./models/download-ggml-model.sh large-v3-turbo-q5_0

D. 疎通テスト

./build/bin/whisper-cli \
  -m ./models/ggml-large-v3-turbo-q5_0.bin \
  -f ./samples/jfk.wav -nt
# → "And so, my fellow Americans..." と表示されれば OK

E. shell script 4 本 + orchestrator

bin/01_ingest.sh 〜 bin/run_all.sh を実装。 ~/.claude/skills/session-recap/SKILL.md の dev_journal INSERT 仕様を参照しながら schema を揃える。

F. claude CLI 整形

claude -p --output-format json \
  --append-system "$(cat prompts/summarize.md)" \
  < work/$ID.raw.txt > out/$ID.payload.json

G. LaunchAgent 登録

launchctl bootstrap gui/$UID \
  ~/Library/LaunchAgents/net.masatoman.voice-pipeline.plist
launchctl enable gui/$UID/net.masatoman.voice-pipeline

H. Shortcuts.app オートメーション

GUI で 1 回設定: 「Voice Memos に新規録音が追加されたら ~/home_work/portfolio/tools/voice-pipeline/inbox/ にコピー」。 これで iPhone 録音も iCloud 経由で inbox に着地。

07ハマりポイントとフォールバック

ハマり 3 大ポイント

  1. Voice Memos の保存先がサンドボックス保護: ~/Library/Group Containers/group.com.apple.VoiceMemos.shared/ は launchd 直接監視で権限エラー → Shortcuts.app 経由で inbox/ にコピーする中継が必須。
  2. m4a を whisper.cpp に直食わせると不安定: ffmpeg -i in.m4a -ac 1 -ar 16000 -c:a pcm_s16le out.wav で 16kHz mono PCM 正規化必須。 01_ingest.sh の唯一の責務。
  3. 日本語句読点が入らない / 固有名詞誤認 (北原・たましん・GビズID 等): --prompt "以下は日本語商談の音声です。 句読点を適切に。 固有名詞: 北原 / たましん / 補助金 / 商工会議所 / アトツギ / DX 認定 / IT 導入補助金 / 都 DX 推進助成金" を whisper.cpp に initial prompt として渡すと体感 2 割改善。 + Voice Memos の Apple Intelligence transcript を Plan B として claude に diff 補正させる。

フォールバック階層

失敗箇所残るデータ復旧手順
whisper.cpp 落ちるdone/ に .m4a + Voice Memos 内 transcript手動で whisper-cli を回し直す。 完全失敗時は Apple Intelligence transcript をコピーして手動 INSERT
claude -p が JSON 壊すwork/$ID.raw.txt + out/$ID.payload.json (壊れた状態)JSON を手動編集 → 04_insert.sh だけ再実行
Supabase INSERT 失敗out/$ID.payload.json (完全)後で session-recap skill 経由で流す or psql で手動 INSERT
各段の中間ファイルが残るので、 どこで失敗しても録音 .m4a は絶対に失われない。 商談の一次情報は何があっても保持される設計。

09既知の課題

  • Intel Homebrew + Rosetta build のため M4 Pro ARM ネイティブ性能の 60-80%。 7.6x realtime で実用十分だが、 もし大量バックログ処理が必要になったら ARM cmake 化検討 (cmake.org から ARM macOS 用 cmake を DL → 別 build dir でリビルド)
  • say コマンドが Bash tool sandbox で動かない: say -v Kyoko -o file.aiff が無声で失敗。 日本語疎通検証は iPhone ボイスメモの実音声でやる方針に変更
  • Core ML 化未着手: 現状 Metal-only。 Core ML 化すると encode 段で 2-3 倍速くなる可能性 (要 Python coremltools)。 速度に不満が出たら検討
  • 話者分離 (diarization) 未対応: whisper.cpp + WhisperX で話者ラベル付与可能だが現状未導入。 1 対 1 の商談なら不要、 3 人以上の場で必要になったら追加
